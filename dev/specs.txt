======================================================================
0. DOCUMENT PURPOSE
======================================================================

This document defines the required behavior, architecture, data model,
user flows, constraints, and release scope for a local-first outbound
marketing application for service businesses.

The product is a desktop-launched local web application that runs on
the user's machine, opens in the browser, stores data locally, uses
external APIs where needed, and orchestrates cold outbound email
campaigns for high-value service sales.

Primary business use case:
- sell AI consulting and similar service offers to profitable SMBs

Target verticals:
- law firms
- car dealerships
- medical practices and clinics
- manufacturing companies
- other high-profit local and regional SMBs

Primary operator:
- a single internal user
- no team collaboration required in v1
- no multi-tenant SaaS behavior in v1

This document intentionally favors practicality, portability, clean
architecture, and strong operational constraints over unnecessary
breadth.

======================================================================
1. PRODUCT DEFINITION
======================================================================

1.1 Product Summary

The application is a local control-plane for outbound email marketing.
It does not attempt to be a full CRM or a full email infrastructure
provider. It owns the logic, workflow state, local database, campaign
generation, scheduling, personalization, reply classification, and
analytics. It delegates lead sourcing and message delivery to external
providers through pluggable adapters.

1.2 Core Product Thesis

The product must allow a single operator to:
- import leads from Apollo and internal lists
- deduplicate and segment leads
- generate outbound campaigns using structured templates plus AI
- preview all scheduled messages before launch
- push campaigns to a sending provider
- monitor sends, replies, unsubscribes, and conversions
- optionally auto-handle constrained reply conversations
- drive prospects to book calls through Google Calendar booking pages

1.3 What The Product Is

The product is:
- a local-first outbound sales workstation
- a browser-based app launched from the local filesystem
- a campaign generator and orchestrator
- a provider-agnostic control plane
- a single-user internal operations tool

1.4 What The Product Is Not

The product is not:
- a full CRM replacement
- a newsletter platform
- a mass consumer marketing platform
- a general web scraping framework
- a raw SMTP infrastructure layer in v1
- an autonomous unrestricted AI sales agent
- a multi-tenant SaaS in v1

======================================================================
2. PRIMARY DESIGN PRINCIPLES
======================================================================

2.1 Local-First
The application UI, backend, worker, and database must run on-device.
The application may call external APIs for lead data, LLM generation,
email delivery, and calendar links.

2.2 Portable
The application folder must be movable. The user must be able to drag
the app folder anywhere and run it from there. No hardcoded absolute
paths are allowed.

2.3 Browser-Based UX
The application should behave like a modern web app, but run locally.
The preferred launch behavior is:
- user double-clicks launcher
- backend starts
- worker starts
- frontend becomes available on localhost
- default browser opens automatically

2.4 Clean Control Plane
The application must own workflow state and business logic. Email
sending, lead sourcing, and LLM generation must be implemented behind
adapters so providers can be swapped later without rewriting the system.

2.5 Human-Inspectable Automation
All outbound content must be inspectable. The user must be able to see:
- what will be sent
- when it will be sent
- through which sending account
- using which template
- with which AI-generated sections

2.6 Moderate But Real Automation
The product should automate the repetitive parts of outbound, but must
not behave recklessly. It must include unsubscribe handling, suppression
logic, send caps, quiet hours, and constrained reply behavior.

2.7 Lean Scope
Only build features that materially improve outbound execution, safety,
or operator efficiency.

======================================================================
3. RESOLVED PRODUCT DECISIONS
======================================================================

The following design questions are considered resolved:

3.1 Use Case
The product is built for service business outbound, especially AI
consulting and adjacent service offers.

3.2 Customer Type
Target buyers are small to medium profitable local/regional businesses.

3.3 Volume Strategy
The system must be architected to support 1,000+ emails/day across
multiple sending accounts, but initial operational defaults should be
more conservative.

3.4 Lead Sources
The system must support:
- Apollo import
- CSV/XLSX import
- persistent local master lead list

3.5 Sending Strategy
The system should use a sending provider adapter in v1 rather than raw
SMTP or raw Gmail sending logic.

3.6 Reply Handling
The system should support:
- threaded follow-ups where possible
- reply classification
- auto-stop on reply
- constrained auto-replies in later release tiers
- human escalation when confidence is low or content is sensitive

3.7 Booking
Meeting booking is not handled by the app calendar engine. Customers
book through a Google Calendar appointment scheduling link.

3.8 Deployment Model
The application is local-first, but the architecture must allow later
deployment to a VM or remote server with minimal change.

3.9 Productization Model
v1 is a single-user internal tool. Multi-tenant SaaS is out of scope.

5. SYSTEM OVERVIEW
======================================================================

5.1 High-Level Components

A. Frontend
- local browser-based web UI
- React + TypeScript recommended
- communicates only with local backend API

B. Backend API
- local HTTP server
- handles CRUD, orchestration logic, settings, campaign generation,
  provider adapter calls, state transitions, analytics aggregation

C. Background Worker
- scheduling
- job execution
- provider polling and webhook processing
- reply classification
- queued task execution
- retry management

D. Local Database
- SQLite in WAL mode recommended for v1
- stores all product state and history

E. Local Filesystem Storage
- logs
- exports
- provider payload snapshots
- generation artifacts
- config files where appropriate
- optional local cache

F. Provider Adapters
- Apollo adapter
- sending provider adapter
- LLM provider adapter
- Google Calendar link adapter
- optional email verification adapter

5.2 Architectural Rule

All provider interactions must go through adapter interfaces. Business
logic must not directly depend on any specific provider SDK or payload
shape beyond adapter boundaries.

11. WORKFLOW SPECIFICATIONS
======================================================================

11.1 Lead Import Workflow

1. User selects import source.
2. System validates file or provider credentials.
3. System reads raw records.
4. System normalizes fields to internal schema.
5. System runs dedupe pass.
6. System produces import result summary:
   - total read
   - inserted
   - merged
   - skipped
   - conflicted
7. System writes audit event.
8. User can review flagged conflicts.

11.2 Campaign Build Workflow

1. User selects audience or filtered leads.
2. User selects offer profile.
3. User selects vertical playbook.
4. User selects template pack.
5. User selects provider and account pool.
6. User configures schedule, quiet hours, approval mode, and reply mode.
7. System renders queue candidates.
8. System performs validation pass:
   - missing email
   - suppression status
   - missing booking link if required
   - missing footer
   - provider account availability
9. System generates queue preview.
10. User approves, edits, regenerates, or cancels.
11. On approval, queue becomes launchable.

11.3 Send Execution Workflow

1. Scheduler finds eligible queued messages.
2. Scheduler rechecks suppression and stop conditions.
3. Scheduler dispatches through provider adapter.
4. System stores provider mapping and status.
5. System writes sent event on success.
6. Failures are recorded with reason and retry behavior.
7. Metrics update.

11.4 Reply Handling Workflow

1. Provider event or polling identifies new reply.
2. System stores raw event.
3. System resolves thread and lead.
4. System classifies reply.
5. System applies policy rules:
   - suppress
   - stop sequence
   - draft reply
   - escalate
6. If auto-reply allowed:
   - wait configured delay
   - send through thread-aware method if available
7. Log all actions.

11.5 Unsubscribe Workflow

1. System detects explicit unsubscribe event or provider flag.
2. System creates suppression entry.
3. System cancels remaining queued steps for that lead.
4. System logs the suppression source.
5. Suppression propagates across future campaigns.

12. AI BEHAVIOR SPECIFICATION
======================================================================

12.1 AI Use Cases

AI is allowed for:
- personalization sentence generation
- subject variant generation
- short opener generation
- reply classification
- reply drafting within constraints
- offer-specific copy adjustments

AI is not allowed to:
- invent factual claims
- invent legal or medical guarantees
- fabricate customer results
- fabricate case studies
- make unsupported compliance claims
- freestyle unrestricted sales negotiation
- generate unsafe regulated advice

12.2 Prompting Principles

The application must use structured prompt templates. Prompts must be
versioned and attributable.

Prompt inputs should include:
- offer profile
- vertical playbook
- lead/company context
- allowed claim list
- banned claim list
- max token/length target
- response schema

12.3 Validation Layer

Generated content must pass validation rules before approval:
- no banned phrases
- no unsupported guarantees
- no invalid missing variables
- no over-length content
- no malformed output schema

12.4 Human Override

The operator must be able to:
- regenerate AI output
- edit final text
- disable AI block usage
- lock template behavior to deterministic mode

- configured relative path

NFR-P3.
The launcher must work regardless of where the folder is placed.

14.3 Performance

NFR-PE1.
The system should handle tens of thousands of leads locally without UI
collapse.

NFR-PE2.

NFR-PE3.

14.4 Security

NFR-S2.

NFR-S3.

14.5 Maintainability

NFR-M2.

NFR-M3.
Provider adapters must be swappable with minimal ripple effects.

NFR-M4.
The database schema must use versioned migrations.

17. OPERATIONAL SAFETY AND GUARDRAILS
======================================================================

17.1 Required Guardrails

G1.
Do not send to suppressed leads.

G2.
Do not send after explicit unsubscribe.

G3.
Do not continue sequence after human reply unless policy explicitly
permits it.

G4.
Do not auto-reply to legal threats, medical details, contracts, or other
high-risk content.

G5.
Do not allow AI to invent claims.

G6.
Do not exceed per-account daily cap.

G7.
Do not send outside configured quiet hours.

G8.
Do not allow campaigns to launch without required footer elements where
configured.

17.2 Circuit Breakers

The system should support automatic campaign pause or alert conditions:
- bounce rate above threshold
- unsubscribe spike
- provider failure spike
- reply classification failure spike
- repeated send account errors

22. IMPLEMENTATION GUIDANCE
======================================================================

22.1 Most Important Engineering Rule

Do not let provider payloads leak into core business logic. Keep clean
internal models and adapter boundaries.

22.2 Second Most Important Rule

The queue preview is the heart of operator trust. If preview accuracy is
weak, the app fails regardless of other features.

22.3 Third Most Important Rule

Constrain AI aggressively. This product is not a sandbox for open-ended
LLM behavior. It is a bounded outbound operations system.

22.4 Fourth Most Important Rule

Build for restart safety from day one. Outbound tools that duplicate
sends or lose state are worthless.

23. FINAL PRODUCT STATEMENT
======================================================================

Build a local-first outbound orchestration application that owns the
workflow, queue, data, AI generation, and operational control for SMB
service-business cold email. The product must run as a browser-based app
launched from the local machine, integrate with Apollo for lead sourcing
and an external sending provider for campaign execution, preview every
scheduled email before launch, enforce suppression and guardrails, and
support constrained reply workflows and Google Calendar booking links.

The implementation should optimize for:
- portability
- clean architecture
- operator visibility
- moderate automation
- deliverability-aware behavior
- low operational friction
- future extensibility without current bloat
