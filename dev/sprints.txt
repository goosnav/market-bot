Build doctrine

Treat this as a local product platform, not a script pile. The right build order is:

Own the domain model and queue state first
Own preview accuracy second
Own restart safety third
Integrate providers only after the core engine is stable
Do not trust “it worked once” as proof

One important correction to the earlier spec: do not use FastAPI BackgroundTasks as the primary job system. FastAPI’s background tasks are for work that runs after returning a response; for this app you want a dedicated worker/scheduler process with persistent queue state, retries, and restart safety. Also, Smartlead’s current API surface explicitly covers campaigns, sending limits/schedules, and webhooks/replies, Apollo’s API covers people search plus people/company enrichment, and Google Calendar appointment schedules provide shareable booking pages. That combination is why the control-plane architecture is the right first move.

Biggest failure modes to prevent
1. Wrong architecture

If provider logic leaks into the core app, the codebase will rot fast.

2. Preview mismatch

If preview does not match what gets sent, operator trust is dead.

3. Duplicate sends

One duplicate-send incident can poison the whole product.

4. Bad reply automation

If the bot says something reckless to a law firm or medical practice, you create real business damage.

5. Path/install fragility

If the local app only works on one machine in one folder, it failed the core requirement.

6. False speed

If you rush provider integration before queue/state correctness, you will spend months debugging nonsense.

Final recommended build order in one line

Lock architecture → build local shell → build data model → build lead system → build templates/AI → build campaign preview → build scheduler → integrate provider → add reply safety → harden and package.
